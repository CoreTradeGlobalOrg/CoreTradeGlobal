In LegalEngagementRepository.js, subscribeToRiskItems method (lines 191-212):

The issue: When a lawyer adds multiple risk items rapidly, serverTimestamp() creates documents with pending/null timestamps in the local Firestore cache. The onSnapshot with orderBy('createdAt', 'desc') may exclude these docs because their timestamps haven't resolved yet.

Fix: Add `includeMetadataChanges: true` as the second argument to onSnapshot, so the callback fires again when pending writes are confirmed by the server. This ensures all documents are included even if their timestamps are temporarily null in the local cache.

Change the onSnapshot call from:
```javascript
return onSnapshot(
  q,
  (snap) => { ... },
  (error) => { ... }
);
```
To:
```javascript
return onSnapshot(
  q,
  { includeMetadataChanges: true },
  (snap) => { ... },
  (error) => { ... }
);
```

This is the same pattern used by Firebase for handling optimistic writes — the snapshot fires once for the local write (with hasPendingWrites: true) and again when the server confirms (with updated timestamps and correct ordering).
