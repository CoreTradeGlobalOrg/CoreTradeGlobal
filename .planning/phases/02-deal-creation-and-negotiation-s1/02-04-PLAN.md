    **2. Create `sendDealEmail` helper function in functions/index.js:**
    - `async function sendDealEmail(to, subject, htmlBody)` — wraps Resend SDK call
    - Non-blocking: email failure should NOT fail the Cloud Function (wrap in try/catch, log error, continue)
    - From address: `'CoreTradeGlobal <onboarding@resend.dev>'` (dev phase — switch to custom domain before production)
    - Follow Pattern 5 from 02-RESEARCH.md

    **3. Create `sendDealNotifications` helper function in functions/index.js:**
    - `async function sendDealNotifications(dealId, eventType, senderUid, deal)` — orchestrates all 3 notification channels
    - Determines recipients: both deal participants except the sender
    - For each recipient:

    **a) Firestore notification document:**
    - Create doc in `users/{recipientId}/notifications/` with:
      - `type: 'deal'`, `eventType` (new_deal, counter_offer, accepted, rejected, expired, withdrawn), `title`, `body`, `dealId`, `dealProductName: deal.productName`, `isRead: false`, `createdAt: Timestamp.now()`, `link: /deals/${dealId}`
    - This feeds the existing notification bell

    **b) FCM push notification (with smart suppression):**
    - Check recipient's `viewingDealId` and `viewingDealSince` fields in their user doc
    - If `viewingDealId === dealId AND viewingDealSince > now - 60000ms` -> SKIP FCM push (user is actively viewing)
    - Otherwise, send FCM data-only message (same pattern as existing `sendMessageNotification`):
      - `data: { type: 'deal_event', dealId, eventType, click_action: /deals/${dealId} }`
    - Handle invalid FCM tokens: clean up if `messaging.send` throws `messaging/invalid-registration-token` or `messaging/registration-token-not-registered`

    **c) Resend email notification:**
    - Build HTML email with deal context: product name, event description, link to deal page
    - Subject lines by event type: "New offer on [product]", "Counter-offer received on [product]", "Offer accepted on [product]", "Offer rejected on [product]", "Offer expired on [product]", "Offer withdrawn on [product]"
    - Email body: brief description + "View Deal" button linking to `/deals/${dealId}`
    - Get recipient email from their user doc

    **4. Create `onDealOfferCreated` trigger in functions/index.js:**
    - `exports.onDealOfferCreated = onDocumentCreated('deals/{dealId}/offers/{offerId}', async (event) => { ... })`
    - Read the new offer doc and the parent deal doc
    - Determine event type based on context: if offer.round === 1, it's 'new_deal'; otherwise 'counter_offer'
    - Call `sendDealNotifications(dealId, eventType, offer.submittedBy, deal)`
    - Also handle sending system message to conversation: write a message to `conversations/{conversationId}/messages/` with content like "New offer submitted on [productName]" or "Counter-offer submitted" with a link to the deal. Use the same message format as existing system messages if the pattern exists.

    **5. Create `onDealStatusChanged` trigger in functions/index.js:**
    - `exports.onDealStatusChanged = onDocumentUpdated('deals/{dealId}', async (event) => { ... })`
    - Compare before.status and after.status
    - If status changed to 'accepted', 'rejected', 'withdrawn', or 'expired':
      - Call `sendDealNotifications(dealId, after.status, determineActor, deal)`
      - The "actor" for accepted/rejected = whoever changed it (the currentTurnUid from the BEFORE snapshot). For expired = 'system'. For withdrawn = the sender of the withdrawn offer.
    - Ignore status changes that don't match terminal states (avoid double-firing)
    - Note: 'accepted' notification here is for both parties. Phase 3 will independently trigger on this status for contract generation.

    **6. Add `renewOffer` Cloud Function:**
    - `exports.renewOffer = onCall(async (request) => { ... })`
    - Auth + accepts `{ dealId, offerId, newExpiryHours }`
    - `db.runTransaction`:
      - Read deal + offer
      - Guard: `offer.submittedBy === uid` (only sender can renew)
      - Guard: `offer.status === 'expired'` (can only renew expired offers)
      - Update offer: status back to 'open', new expiresAt = now + newExpiryHours
      - Update deal: status back to 'negotiating', currentTurnUid = other party, updatedAt = now
    - Returns `{ success: true }`
    - Post-transaction: send notifications about renewed offer

    **1. Create `checkExpiredOffers` scheduled Cloud Function in functions/index.js:**
    - `exports.checkExpiredOffers = onSchedule({ schedule: 'every 30 minutes', timeZone: 'UTC', retryCount: 3 }, async () => { ... })`
    - Query: `db.collectionGroup('offers').where('status', '==', 'open').where('expiresAt', '<=', Timestamp.now())`
    - For each expired offer:
      - Get parent deal ID: `offerDoc.ref.parent.parent.id`
      - Use a batched write to update both offer status to 'expired' and deal status to 'expired' (batch, not transaction, since we're processing multiple deals)
    - If batch is non-empty, commit
    - Log: "Expired N offers across M deals"
    - NOTE: The composite index for `collectionGroup('offers')` on `status` + `expiresAt` was already added in Plan 01's firestore.indexes.json

    **2. Create `sendExpiryReminders` scheduled Cloud Function (or add to checkExpiredOffers):**
    - Run alongside or within the checkExpiredOffers function
    - Query open offers approaching expiry at 3 reminder windows:
      - 24h window: expiresAt between now+23h and now+24h30m (30min buffer matches schedule interval)
      - 4h window: expiresAt between now+3h30m and now+4h30m
      - 1h window: expiresAt between now+30m and now+1h30m
    - For each offer in a reminder window:
      - Check `remindersSet` array field on the offer (e.g., `['24h']`). If the relevant reminder already sent, skip.
      - Update offer doc to add the reminder level to `remindersSet` array
      - Send notification + email to the receiver (the party whose turn it is = deal.currentTurnUid):
        - Subject: "Offer expiring in [time] on [product]"
        - Body: "The current offer on [product] expires in approximately [time]. Take action before it expires."
      - Also send to the sender as an FYI: "Your offer on [product] expires in [time]"
    - Use `arrayUnion` for updating `remindersSet` to avoid race conditions

    **3. Update Navbar to include "My Deals" link for members:**
    - Read the existing Navbar component to understand the navigation link structure
    - Add a "My Deals" navigation item visible to all authenticated users (members, and admin via admin override pattern from Phase 1)
    - Use Lucide `Handshake` icon (or `FileText` if Handshake is not available in the installed version)
    - Link to `/deals`
    - Position: near Messages or after existing member links
    - Follow existing pattern for member-visible links (check how other links like Messages, Dashboard are conditionally shown)
