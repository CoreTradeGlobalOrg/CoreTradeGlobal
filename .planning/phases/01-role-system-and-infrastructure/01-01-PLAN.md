**2. Add Cloud Functions to `functions/index.js`:**

Read the existing file first to understand current patterns (uses `onCall`, `admin`, `db`, `isUserAdmin` helper).

**`inviteUser` (onCall):**
- Accepts: `{ email, role, name, company }`
- Validates: caller is admin (use existing `isUserAdmin` pattern), role is in valid invite roles
- Creates Firebase Auth user with `admin.auth().createUser({ email, displayName: name })`
- Sets custom claims: `admin.auth().setCustomUserClaims(uid, { role })`
- Creates Firestore `users/{uid}` doc with fields: email, displayName, companyName, role, inviteStatus: 'pending', invitedBy, invitedAt, createdAt, emailVerified: false, adminApproved: true
- Creates Firestore `invites/{uid}` doc with: email, role, name, company, status: 'pending', invitedBy, invitedAt, expiresAt (7 days), expireAt (same as expiresAt — TTL field)
- Generates sign-in link: `admin.auth().generateSignInWithEmailLink(email, { url: APP_URL/onboarding?uid=UID, handleCodeInApp: true })`
- Stores the generated link in the invite doc (for resend capability)
- Returns `{ success: true, uid }`
- Error handling: if createUser fails (email exists), throw appropriate HttpsError

**`setUserRole` (onCall):**
- Accepts: `{ userId, role }`
- Validates: caller is admin, role is one of the 5 valid roles
- Sets custom claims: `admin.auth().setCustomUserClaims(userId, { role })`
- Updates Firestore `users/{userId}` doc: `{ role }`
- Returns `{ success: true }`
- This replaces the direct Firestore write in `handleToggleAdmin`

**`migrateExistingUsers` (onCall):**
- Admin-only callable function
- Queries all users from Firebase Auth using `admin.auth().listUsers()`
- For each user without a `role` custom claim, sets `{ role: 'member' }` via `setCustomUserClaims`
- Also checks Firestore `users` collection: if a user has `role: 'admin'` in Firestore, sets their claim to `admin` instead of `member`
- Returns `{ migrated: count, skipped: count }`
- This is a one-time migration function for bootstrapping existing accounts

Use the APP_URL from `process.env.APP_URL` or `functions.config().app.url` for the invite link. Add a comment noting the TTL policy for the `invites` collection should be configured in Firebase Console (Collection=invites, Field=expireAt).

**1. Update `src/lib/firebase-admin.js`:**

Read the existing file. Currently `verifyIdToken` returns `{ valid, uid, email }`. Update it to also return `role` from the decoded token's custom claims: `role: decodedToken.role || null`. This makes the role available to the session API without additional lookups.

**2. Update `src/app/api/auth/session/route.js`:**

Read the existing file. The current POST handler takes `role` from the request body — this is the security gap noted in the codebase comments. Fix it:

- After calling `verifyIdToken(idToken)`, use the returned `role` field (from custom claims) instead of `request body role`
- If `role` is null/undefined (existing member without claims), default to `'member'`
- Set the session cookie with the role from the verified token: `JSON.stringify({ uid, email, role: verifiedRole, verified: true })`
- Add a comment: `// Role from verified ID token custom claims — not from client body (security fix)`
- Keep all other existing session logic (rate limiting, cookie options, DELETE handler) unchanged

**3. Rewrite `firestore.rules`:**

Read the existing file to understand current structure. Rewrite with custom claims pattern:

- Add helper functions at the top:
  - `function isAuthenticated() { return request.auth != null; }`
  - `function userRole() { return request.auth.token.role; }`
  - `function isAdmin() { return isAuthenticated() && userRole() == 'admin'; }`
  - `function isMember() { return isAuthenticated() && (userRole() == 'member' || userRole() == null); }` — null fallback for existing accounts per CONTEXT.md
  - `function isProvider() { return isAuthenticated() && (userRole() == 'logistics_provider' || userRole() == 'insurance_provider'); }`
  - `function isInsuranceProvider() { return isAuthenticated() && userRole() == 'insurance_provider'; }`
  - `function isLogisticsProvider() { return isAuthenticated() && userRole() == 'logistics_provider'; }`
  - `function isLawyer() { return isAuthenticated() && userRole() == 'lawyer'; }`
  - `function isOwner(userId) { return isAuthenticated() && request.auth.uid == userId; }`

- Preserve all existing collection rules but replace any `get()` calls for role checking with the custom claims helpers
- Keep the admin global override: `match /{document=**} { allow read, write: if isAdmin(); }`
- For `users/{userId}`: allow read if isAuthenticated(), allow write if isOwner(userId) BUT block writing `role` field (use `!('role' in request.resource.data) || request.resource.data.role == resource.data.role` to prevent role self-modification), allow full write if isAdmin()
- For `invites/{inviteId}`: allow read if isAdmin(), allow write if isAdmin()
- Keep existing rules for products, requests, messages, conversations, categories, fairs, news — just replace role checks from doc reads to custom claims
- Add comments explaining the null-role fallback for existing member accounts

Ensure the rules still allow all currently-working functionality (product browsing, RFQs, messaging, admin panel) while adding the new role enforcement layer.
