    **Fix 4 -- Session error visibility and security cleanup (UAT gap 3, blocker):**

    The middleware route protection logic is correct -- it properly checks `providerRoutes` and `lawyerRoutes` against the session cookie role. The UAT failure ("member can navigate to /provider") happens because the session cookie either has no role data or was never created due to a silent session API failure (most likely from a missing `FIREBASE_SERVICE_ACCOUNT_KEY`).

    The user must configure `FIREBASE_SERVICE_ACCOUNT_KEY` in `.env.local` (handled by user_setup checkpoint below). However, the code should surface failures clearly rather than swallowing them silently.

    In `src/presentation/contexts/AuthContext.jsx`, replace the session cookie block (lines 80-93) with:

    ```js
    // Set session cookie for middleware authentication
    // SECURITY: Send Firebase ID token for server-side verification
    try {
      const idToken = await firebaseUser.getIdToken();
      const sessionResponse = await fetch('/api/auth/session', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ idToken }),
      });
      if (!sessionResponse.ok) {
        console.warn(
          `Session cookie not set (status ${sessionResponse.status}). ` +
          'Role-based route protection may not work. ' +
          'Check FIREBASE_SERVICE_ACCOUNT_KEY in .env.local.'
        );
      }
    } catch (cookieError) {
      console.error('Failed to set session cookie:', cookieError);
    }
    ```

    Key changes from the current code:
    1. **Remove `role: userData.role` from request body.** The session API already reads role from the verified JWT token (security fix from 01-01). Sending role from the client is the old insecure pattern. Change `body: JSON.stringify({ idToken, role: userData.role })` to `body: JSON.stringify({ idToken })`.
    2. **Store fetch response and check `.ok`.** When the session API returns non-200 (e.g., 401 because FIREBASE_SERVICE_ACCOUNT_KEY is missing), log a clear console warning naming the env var so developers can diagnose the issue immediately.
  </action>

  </verify>
